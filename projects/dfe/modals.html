Modals

How to use

When to use:
  • Important warnings
  • Preventing critical errors
  • Requesting input of info critical to process/workflow
  • Fragmenting complicated workflow into simpler steps
  • Asking user for information that could lessen effort/work
When not to use:
  • If an error/success/warning can be related to a form field or within the actual UI
  • If the modal has a complicated user flow that takes user out of the original page
  • Gathering information that is not related to the workflow process
For using buttons within modals:
  • Use two buttons of equal weight (one filled and one outline) if the actions themselves are of equal importance to the user
  • Use two buttons of unequal weight (one filled and one text button) if there is one obvious choice for a user action
  • Primary action should always be on the left, secondary action should be on the right

Modal Types

Confirmation Modal

  • Forcing the user to complete something outside of their normal workflow
  • Happens after a user action on a page
  • Confirming things that can't be undone

Warning Modal

  • Type of confirmation modal
  • Happens after user action on a page
  • Confirming things that can't be undone

Acknowledgement Modal

  • Shouldn't be able to close without making a decision or acknowledging something
  • Close button disabled

System Error Modal

  • Type of acknowledgement modal but with the ability to close out
  • Used to communicate a technical error the system is experiencing
  • No user action taken
  • These are typically default messages from the browser

User Error Modal

  • Type of acknowledgement modal
  • Used to communicate an error that was triggered by user action
